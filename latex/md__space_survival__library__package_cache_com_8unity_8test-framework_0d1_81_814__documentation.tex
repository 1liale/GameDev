When writing tests, it is possible to avoid duplication of code by using the \href{https://github.com/nunit/docs/wiki/SetUp-and-TearDown}{\texttt{ Set\+Up and Tear\+Down}} methods built into \href{http://www.nunit.org/}{\texttt{ N\+Unit}}. The \mbox{\hyperlink{namespace_unity}{Unity}} Test Framework has extended these methods with extra functionality, which can yield commands and skip frames, in the same way as /\+Users/ray/\+Documents/\+Git\+Hub/\+Game\+Dev/\+Space\+Survival/\+Library/\+Package\+Cache/com.unity.\+test-\/framework@1.\+1.\+14/\+Documentation$\sim$/reference-\/attribute-\/unitytest.md \char`\"{}\+Unity\+Test\char`\"{}.\hypertarget{md__space_survival__library__package_cache_com_8unity_8test-framework_0d1_81_814__documentation_autotoc_md482}{}\doxysection{Action execution order}\label{md__space_survival__library__package_cache_com_8unity_8test-framework_0d1_81_814__documentation_autotoc_md482}
The actions related to a test run in the following order\+:


\begin{DoxyItemize}
\item Attributes implementing \href{https://github.com/nunit/docs/wiki/IApplyToContext-Interface}{\texttt{ I\+Apply\+To\+Context}}
\item Any attribute implementing \href{\#outerunitytestaction}{\texttt{ Outer\+Unity\+Test\+Action}} has its {\ttfamily Before\+Test} invoked
\item Tests with \href{\#unitysetup-and-unityteardown}{\texttt{ Unity\+Set\+Up}} methods in their test class.
\item Attributes implementing \href{https://github.com/nunit/docs/wiki/ICommandWrapper-Interface}{\texttt{ I\+Wrap\+Set\+Up\+Tear\+Down}}
\item Any \href{https://github.com/nunit/docs/wiki/SetUp-and-TearDown}{\texttt{ Set\+Up}} attributes
\item \href{https://nunit.org/docs/2.6/actionAttributes.html}{\texttt{ Action attributes}} have their {\ttfamily Before\+Test} method invoked
\item Attributes implementing of \href{https://github.com/nunit/docs/wiki/ICommandWrapper-Interface}{\texttt{ I\+Wrap\+Test\+Method}} ~\newline

\item {\bfseries{The test itself runs}}
\item \href{https://nunit.org/docs/2.6/actionAttributes.html}{\texttt{ Action attributes}} have their {\ttfamily After\+Test} method invoked
\item Any method with the \href{https://github.com/nunit/docs/wiki/SetUp-and-TearDown}{\texttt{ Tear\+Down}} attribute
\item Tests with \href{\#unitysetup-and-unityteardown}{\texttt{ Unity\+Tear\+Down}} methods in their test class
\item Any \href{\#outerunitytestaction}{\texttt{ Outer\+Unity\+Test\+Action}} has its {\ttfamily After\+Test} invoked
\end{DoxyItemize}

The list of actions is the same for both {\ttfamily Test} and {\ttfamily Unity\+Test}.\hypertarget{md__space_survival__library__package_cache_com_8unity_8test-framework_0d1_81_814__documentation_autotoc_md483}{}\doxysection{Unity\+Set\+Up and Unity\+Tear\+Down}\label{md__space_survival__library__package_cache_com_8unity_8test-framework_0d1_81_814__documentation_autotoc_md483}

The presence of this attribute causes the {\bfseries{Test Runner}} to expect every single log. By default, the Test Runner only fails on error logs, but {\ttfamily Test\+Must\+Expect\+All\+Logs} fails on warnings and info level messages as well. {\ttfamily Unity\+Test} attribute is the main addition to the standard \href{http://www.nunit.org/}{\texttt{ N\+Unit}} library for the \mbox{\hyperlink{namespace_unity}{Unity}} Test Framework. This type of unit test allows you to skip a frame from within a test (so background tasks can finish) or give certain commands to the \mbox{\hyperlink{namespace_unity}{Unity}} {\bfseries{Editor}}, such as performing a domain reload or entering {\bfseries{Play Mode}} from an {\bfseries{Edit Mode}} test.

In Play Mode, the {\ttfamily Unity\+Test} attribute runs as a \href{https://docs.unity3d.com/Manual/Coroutines.html}{\texttt{ coroutine}}. Whereas Edit Mode tests run in the \href{https://docs.unity3d.com/ScriptReference/EditorApplication-update.html}{\texttt{ Editor\+Application.\+update}} callback loop.

The {\ttfamily Unity\+Test} attribute is, in fact, an alternative to the {\ttfamily N\+Unit} \href{https://github.com/nunit/docs/wiki/Test-Attribute}{\texttt{ Test attribute}}, which allows yielding instructions back to the framework. Once the instruction is complete, the test run continues. If you {\ttfamily yield return null}, you skip a frame. That might be necessary to ensure that some changes do happen on the next iteration of either the {\ttfamily Editor\+Application.\+update} loop or the \href{https://docs.unity3d.com/Manual/ExecutionOrder.html}{\texttt{ game loop}}.\hypertarget{md__space_survival__library__package_cache_com_8unity_8test-framework_0d1_81_814__documentation_autotoc_md502}{}\doxysection{Edit Mode example}\label{md__space_survival__library__package_cache_com_8unity_8test-framework_0d1_81_814__documentation_autotoc_md502}

A test fails if \mbox{\hyperlink{namespace_unity}{Unity}} logs a message other than a regular log or warning message. Use \href{\#logassert}{\texttt{ Log\+Assert}} to check for an expected message in the log so that the test does not fail when \mbox{\hyperlink{namespace_unity}{Unity}} logs the message.

Use {\ttfamily Log\+Assert.\+Expect} before running the code under test, as the check for expected logs runs at the end of each frame.

A test also reports a failure, if an expected message does not appear, or if \mbox{\hyperlink{namespace_unity}{Unity}} does not log any regular log or warning messages.\hypertarget{md__space_survival__library__package_cache_com_8unity_8test-framework_0d1_81_814__documentation_autotoc_md556}{}\doxysection{Example}\label{md__space_survival__library__package_cache_com_8unity_8test-framework_0d1_81_814__documentation_autotoc_md556}

An interface for receiving callbacks when running tests. All test runs invoke the callbacks until the next domain reload.

The {\ttfamily Run\+Started} method runs when the whole test run starts. Then the {\ttfamily Test\+Started} method runs with information about the tests it is about to run on an assembly level. Afterward, it runs on a test fixture level and then on the individual test. If the test is a \href{./https://github.com/nunit/docs/wiki/Parameterized-Tests}{\texttt{ parameterized test}}, then it is also invoked for each parameter combination. After each part of the test tree have completed running, the corresponding {\ttfamily Test\+Finished} method runs with the test result. At the end of the run, the {\ttfamily Run\+Finished} event runs with the test result.

In some cases, it is relevant to perform changes to \mbox{\hyperlink{namespace_unity}{Unity}} or the file system before building the tests. In the same way, it may be necessary to clean up such changes after the test run. In response to such needs, you can incorporate the pre-\/build setup and post-\/build cleanup concepts into your tests in one of the following ways\+:


\begin{DoxyEnumerate}
\item Via implementation of {\ttfamily I\+Prebuild\+Setup} and {\ttfamily I\+Post\+Build\+Cleanup} interfaces by a test class.
\item Via applying the {\ttfamily Prebuild\+Setup} attribute and {\ttfamily Post\+Build\+Cleanup} attribute on your test class, one of the tests or the test assembly, providing a class name that implements the corresponding interface as an argument (fx {\ttfamily \mbox{[}Prebuild\+Setup(\char`\"{}\+My\+Test\+Scene\+Setup\char`\"{})\mbox{]}}).
\end{DoxyEnumerate}\hypertarget{md__space_survival__library__package_cache_com_8unity_8test-framework_0d1_81_814__documentation_autotoc_md601}{}\doxysection{Execution order}\label{md__space_survival__library__package_cache_com_8unity_8test-framework_0d1_81_814__documentation_autotoc_md601}
All setups run in a deterministic order one after another. The first to run are the setups defined with attributes. Then any test class implementing the interface runs, in alphabetical order inside their namespace, which is the same order as the tests run.

\begin{quote}
{\bfseries{Note}}\+: Cleanup runs right away for a standalone test run, but only after related tests run in the \mbox{\hyperlink{namespace_unity}{Unity}} Editor. \end{quote}
